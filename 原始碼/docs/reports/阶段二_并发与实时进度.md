# 阶段二汇报 · 任务队列、并发控制与实时进度

## 背景与目标
平台要给多人同时用,但大模型算力是全研发中心共享、并发有限。若不加控制,多人一起生成会把共享模型挤垮、互相拖慢。本阶段引入**任务队列 + 并发闸**,并提供**实时进度**与**队列可视**。

## 做了什么
1. **Redis 任务队列 + 并发闸**:同时运行 ≤2、等待 ≤2;系统内(运行+等待)达到 4 时,**第 5 个请求被明确拒绝**并提示"排队已满"。三个数都走配置。
2. **任务状态机**:queued → running → done / failed / canceled,可取消;状态以数据库为准,Redis 维护实时队列与进度。
3. **实时进度(SSE)**:生成过程把"当前第几章 / 总章数 / 日志 / 片段 / 完成 / 错误"实时推给前端;**断线可重连补拉**已产生的进度。
4. **队列状态接口**:实时返回运行数、排队数、上限与各任务排位。

## 技术实现
- 新增 `taskqueue.py`:队列、并发闸、容量判定、进度发布、后台调度线程。
- 生成核心抽出 `execute_task`,**支持中途取消**(在分章节与流式输出之间检查取消标记)。
- 新增接口:
  - `POST /api/cases/{id}/generate`(带 `queued=true`)→ 入队,返回 task_id + 排队位置;满则返回 429。
  - `GET /api/tasks/{id}` 查询状态/进度;`GET /api/tasks/{id}/stream` SSE 进度流;`POST /api/tasks/{id}/cancel` 取消。
  - `GET /api/queue/status` 队列总体占用。

## 效果与价值
- 多人并发**零串扰**:并发严格 2、排队 2、满即拒,保护共享模型。
- 生成不再"盲等":实时看到进度,断线能续。
- 算力占用一目了然,便于排查与调度。

## 演示步骤
1. 连续提交多个生成任务(`queued=true`):前 2 个运行、再 2 个排队、第 5 个被拒绝;
2. 用 `GET /api/queue/status` 看运行/排队数字;
3. 订阅 `GET /api/tasks/{id}/stream` 看逐章实时进度;
4. 对排队中的任务调 cancel,状态变为 canceled。

## 验收
并发数严格为 2、队列为 2、满 4 拒绝;状态流转与取消正确;SSE 实时且可重连。自测通过。
