## 阶段一 · 覆盖最大化引擎(Task 8)

**做什么**:在调模型之前插入一层**确定性枚举**,让"测哪些点"由程序决定,模型只负责把每个点填成标准卡片。
- 边界值分析(BVA):每个参数取上下限,生成 下界-误差 / 下界 / 下界+误差 / 正常值 / 上界-误差 / 上界 / 上界+误差;误差按参数类型(电压/电流/温度)走配置。
- 成对覆盖(pairwise):用 allpairspy 对多参数的边界点做成对组合,覆盖所有两两组合而不必全穷举。
- 约束过滤:排除非法/互斥组合(如 min>max)。
- 离线运行:组合工具一律本地,绝不调在线 API。

**汇报话术**:
> 之前测试用例的数量和覆盖完全依赖大模型"自觉",不稳定也不可证明。本阶段引入确定性覆盖引擎:对每个参数按边界值分析自动生成 7 个测试点,再用成对组合算法覆盖参数间的两两交互,并自动过滤非法组合。**测试点的数量和覆盖范围由程序保证、可量化、可复现**,模型只负责把每个点填成规范卡片。这是测试质量从"看模型心情"到"工程可控"的关键一步。

## 阶段二 · 队列并发 + 实时进度(Task 5 / 7 / 12)

**做什么**:
- Redis 任务队列 + 并发闸:同时运行 running≤2、等待 queued≤2、系统满 4 时第 5 个请求拒绝(三个数走配置)。
- 任务状态机:queued→running→done/failed,可 cancel;状态 PG+Redis 双写、以 PG 为准。
- SSE 实时进度:`GET /api/tasks/{id}/stream` 推送 chunk/progress/log/done/error,断线可续。
- 队列状态接口:`GET /api/queue/status` 返回运行数/排队数/上限/各任务排位。

**汇报话术**:
> 本阶段让平台具备多人并发能力:用任务队列和并发闸控制同时只跑 2 个生成、最多 2 个排队,算力占满时明确拒绝并提示,避免多人一起用把共享的大模型挤垮。生成过程通过实时进度推送,用户能看到"第几章/总进度",不再盲等。并提供队列状态查询,占用情况一目了然。

## 阶段三 · 语义检索 + 部署交付(Task 11 / 13)

**做什么**:
- 检索:`GET /api/search?q=` 关键词检索 + pgvector 向量语义检索(BGE-M3 生成向量);嵌入不可用时自动降级为关键词检索。
- 部署:nginx 反代(`/` 托管前端、`/api/*` 反代后端、SSE 关闭缓冲、上传体积上限),部署文档沉淀。

**汇报话术**:
> 收尾阶段让历史成果可被"找回来":支持关键词检索,并基于向量数据库做语义检索——用相近描述也能召回相关的历史用例。同时完成生产部署方案:用 nginx 统一入口反向代理,关闭 SSE 缓冲保证实时性,设置上传上限,严禁裸暴露后端端口;部署步骤文档化,可复用。
